Variables first created in a pre-request script via `pm.variables.set("token", v)` were not substituted into request headers (or url/params/body/auth/form-data) when **no active environment** existed. Predefined env/collection vars worked because they already lived in a read scope. FIX = proper transient **local variable** scope (Postman semantics): (1) `scriptRunner.js` ‚Äî `pm.variables` decoupled from `pm.environment` into a `localUpdates` store seeded from `context.localVariables`; `get` resolves local‚Üíenv‚Üícollection, `set`/`unset` operate on local; `executeScript` now returns `varUpdates` (the authoritative full local map; empty/whitespace early-return preserves the seed); `pm.environment.set` still persists via `envUpdates` unchanged. (2) `substituteVariables.js` ‚Äî `buildEnvMap` accepts `localVariables` set LAST (priority local > environment > collection); `substituteEnv`/`substituteUrl` forward it. (3) `useResponseExecution.js` + `useWorkflowExecution.js` ‚Äî keep a `localVars` accumulator **independent of `currentEnv`**, ASSIGN (not merge) each script's `varUpdates` so `unset` propagates, and thread `localVariables: localVars` into every `executeScript`/`substituteEnv`/`substituteUrl` (url incl. pathVariables, headers, body, auth token, form-data); workflow seeds each step from the root collection pre-script's local vars and rescopes per step. Behavior change: `pm.variables.set` no longer persists to the environment (Postman-correct). NEW regression unit tests `src/utils/scriptRunner.test.js` (12 cases: header resolve with no env [red‚Üígreen], varUpdates exposure, substituteEnv/Url localVariables, precedence, transient non-leak, env.set still persists, get readback, unset propagation, empty-script seed retention, cross-script seed visibility, predefined+local coexistence).
